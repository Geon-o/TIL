# 접근제어자와 캡슐화

> 접근제어자(Access Modifier)
* 객체 지향에서 정보 은닉이란 굳이 알 필요가 없는 정보를 사용자로부터 숨겨야한다는 개념이다. 그렇게 함으로써
사용자는 최소한의 정보만으로 프로그램을 손쉽게 사용할 수 있게 된다.
* Java에서는 이러한 정보 은닉을 위해 접근 제어자라는 기능을 제공하고 있다.
* 접근 제어자를 사용하게 되면 접근제어가 설정된 필드변수를 직접적으로 제어하지 못하고 엑세스가 가능한 메서드를 통해 제어가 가능하다.
<br/>

> 접근제어자 종류
* 접근제어자는 총 4가지의 종류가 있다.
1. public
2. protected
3. (default)
4. private

* 접근제어자는 각각 접근할 수 있는 제한범위를 갖는다.
* public > proteted > (default) > private 순으로 오른쪽으로 갈수록 접근범위가 크게 줄어든다.
<br/>


## public과 private의 차이
> public
* public은 접근범위가 매우 넓다. 즉 어디서든 누구나 자유롭게 액세스 할 수 있고 값의 변경과 참조가 가능해진다.
* 하지만 이런 접근은 다른 사용자에게 불필요한 데이터를 노출시키는 것이기 떄문에 보안에 취약한 점이 있다.

> private
* 그리하여 사용하게 되는 private, private는 부정한 엑세스를 막아 다른 사용자로부터 데이터의 변경이나 제어를 막을 수 있고
프로그램 내부적으로 부정한 값도 막을 수 있게 되는 제어자이다.
* private 제어자를 보호하고 싶은 멤버에 붙이게 되면 자유롭게 엑세스 할 수 없는 이런 구조를 캡슐화라 한다.
<br/>

## 캡슐화
* 접근제어자 private를 사용하여 설정할 수도 있고, 멤버변수나 메서드를 하나로 묶어 외부에서 프로그램으로 엑세스 할때 감추어 은닉하는 걸 말한다.
<br/>

### 그럼 캡슐화를 사용하였을 때 이점은 무엇일까.
1. 코드의 중복을 피할 수 있다.
* 멤버변수, 메서드를 하나로 묶어주면서 객체의 모듈화가 이루어진다. 이렇게 모듈화를 통해 독립성이 높아지면 코드의 중복을 잡을 수 있을 뿐더러
코드를 재사용할 수 있다. 이러한 코드는 유지보수하기에 수월하다.

2. 객체의 구현 정보 은닉
* 외부에서 소스로 엑세스 할때 노출되지 말하야 할 코드가 노출되어 외부의 잘못된 사용으로 객체의 손상을 방지하지 위해 캡슐화를 사용한다.
<br/>

### 그렇다면 캡슐화된 객체를 어떻게 접근해야 할까.
> Getter / Setter 메서드
* 객체의 데이터를 객체의 외부에서 직접적으로 접근하는 것을 막으므로 외부에서 마음대로 읽고 변경할 경우 무결성의 원칙이 깨질 수 있다. 
이런 경우 get메서드와 set메서드를 사용하여 데이터 저장, 수정, 출력을 할 수 있다. 보통 DB 사용 시 유용하게 사용된다.
<br/>

> Getter
* 외부에서 객체의 데이터를 읽을 때 사용되며, 읽어온 데이터는 호출된 곳으로 전달된다.
<br/>

> Setter
* 외부에서 객체의 데이터를 변경, 수정할 때 사용되며, 데이터는 해당 메서드를 통해 전달된다.
